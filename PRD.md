## 1. 需求背景与问题分析

### 1.1 业务背景

Android 车载系统（Android Automotive OS）已成为各大车厂智能座舱的核心操作系统。用户在车载场景中对本地媒体（USB 音乐、U 盘视频、SD 卡图片）的播放需求占比极高：

- 根据车载行业调研，超过 **65%** 的车主习惯通过 USB 设备在车内播放音乐
- 车载 USB 存储容量通常在 **32GB ~ 256GB** 之间，文件量级在 **2000 ~ 50000 个** 不等
- 用户一上车最直观的体验就是：插上 U 盘后能否 **立刻** 听歌

### 1.2 核心痛点

#### 痛点一：原生扫描速度无法接受

| 场景             | 原生 MediaScanner 表现                  | 用户感知         |
| ---------------- | --------------------------------------- | ---------------- |
| USB 3000 个文件  | 扫描耗时 **6 分钟以上**                  | 上车等半天才能放歌 |
| USB 10000 个文件 | 扫描耗时 **15 分钟以上**，期间 UI 可能卡顿 | 完全不可接受     |
| USB 50000 个文件 | 可能出现 ANR，甚至扫描超时失败            | 功能不可用       |

> **根因**: 原生 MediaProvider 采用单线程扫描队列，所有挂载卷的扫描任务串行排队，且逐文件解析元数据的 IO 开销未做优化。

#### 痛点二：车机供应商裁剪不一致

不同车厂/供应商对 Android Framework 层 MediaScannerService 和 MediaProvider 的裁剪程度不同，导致：

- 上层播放器应用在不同车型上行为不一致
- 应用开发方需要为每个车型做适配，**平台迁移成本极高**
- 部分车型甚至在 Framework 层完全移除了媒体扫描能力

#### 痛点三：多 USB 场景无法并行处理

车载场景常见多 USB 同时接入（如：一个音乐 U 盘 + 一个视频 U 盘），原生方案只能串行扫描，后插入的设备需等待前一个设备扫描完成。

### 1.3 解决方案定位

开发一个**系统签名级独立媒体扫描应用**（`/system/priv-app/MediaScanner/`），自建数据库完全替代原生 MediaStore，对外提供兼容原生字段的 ContentProvider + AIDL 双通道 API，生态应用零成本或极低成本接入。

### 4.2 详细功能规格

#### F01: 多存储卷管理

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 系统需管理所有挂载的存储卷（内部存储、USB 设备、SD 卡），为每个卷维护独立的状态与扫描记录 |
| **输入**  | 系统存储广播（`MEDIA_MOUNTED` / `MEDIA_UNMOUNTED` / `MEDIA_EJECT` / `MEDIA_BAD_REMOVAL`） |
| **处理**  | 1. 识别卷类型（INTERNAL / USB / SDCARD）<br>2. 创建 Volume 对象并持久化数据库<br>3. 维护卷状态机（MOUNTED → SCANNING → READY / FAILED）<br>4. 去重 - 相同路径不重复注册 |
| **输出**  | Volume 列表通过 ContentProvider URI `content://.../storage` 可查询 |
| **异常处理** | 异常拔出（`MEDIA_BAD_REMOVAL`）: 清理该卷缓存数据，记录异常日志 |

#### F02: 全量扫描引擎

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 广度优先遍历目标目录树，识别并解析所有支持的媒体文件          |
| **支持格式** | 音频: MP3 / AAC / FLAC / WAV / OGG / WMA<br>视频: MP4 / AVI / MKV / MOV / WMV / FLV<br>图片: JPEG / PNG / BMP / GIF / WebP / HEIF |
| **元数据字段** | 音频: title / artist / album / genre / duration / bitrate / sampleRate / channels<br>视频: title / duration / width / height / frameRate / rotation<br>图片: width / height / mimeType / orientation |
| **遍历策略** | BFS 广度优先，自动跳过隐藏目录、缓存目录（`.thumbnails`、`Android`、`cache` 等） |
| **解析超时** | 单文件解析超时 5 秒，超时则跳过并记录 warning 日志            |
| **批量写入** | 每 100 条记录批量写入数据库，减少 IO 次数                     |

#### F03: 增量扫描

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 基于文件修改时间戳（`date_modified`），仅处理变化的文件，大幅降低重复扫描开销 |
| **三方比对逻辑** | - **新增**: 磁盘存在, DB不存在 → 解析 + 插入<br>- **修改**: 磁盘存在, DB存在, mtime不同 → 重新解析 + 更新<br>- **无变化**: 磁盘存在, DB存在, mtime相同 → 跳过（占大多数）<br>- **已删除**: 磁盘不存在, DB存在 → 从DB删除 |
| **触发时机** | 1. 应用启动时检查所有 READY 状态的卷<br>2. 定时任务（默认 30 分钟间隔）<br>3. 用户通过 AIDL 接口手动触发 |

#### F04: USB/SD 热插拔响应

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **插入处理** | 1. 识别卷并创建 Volume（状态 → MOUNTED）<br>2. 自动创建全量扫描任务（优先级 = VOLUME_MOUNTED）<br>3. 入队扫描，开始遍历解析<br>4. 完成后 Volume 状态 → READY |
| **拔出处理** | 1. 查找对应 Volume<br>2. 取消正在进行的扫描任务<br>3. **数据库事务**: 删除该卷所有媒体记录 → 更新 Volume 状态为 REMOVED<br>4. 通过 ContentProvider `notifyChange()` 通知客户端刷新 |
| **性能要求** | 插入至扫描开始 < 3 秒；拔出至数据清除 < 5 秒                  |

#### F05: 开机恢复

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 设备重启后自动初始化数据库，恢复各卷的上次扫描状态，确保用户无需等待重新全量扫描 |
| **流程**  | 1. 接收 `BOOT_COMPLETED` 广播<br>2. 初始化 Room Database<br>3. 通过 `StorageManagerService.getStorageVolumes()` 获取当前挂载卷<br>4. 与 DB 中 Persistent Volume 对比（新卷触发扫描 / 已有卷恢复 READY / 已拔出卷标记 REMOVED）<br>5. 恢复未完成的扫描任务 |

#### F06: ContentProvider API

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **Authority** | `com.txzing.media.scanner.provider`                          |
| **URI 设计** | `content://.../media` - 所有媒体<br>`content://.../media/type/{audio\|video\|image}` - 按类型过滤<br>`content://.../media/{id}` - 按 ID 查询<br>`content://.../storage` - 存储卷列表<br>`content://.../scan_tasks` - 扫描任务列表 |
| **查询参数** | `type`、`volume_id`、`sort`、`min_duration`、`min_width`、`min_height` |
| **权限保护** | `signature` 级别，仅同签名应用可访问                          |

#### F07: AIDL 扫描控制 API

| 接口方法               | 描述                           |
| ---------------------- | ------------------------------ |
| `scan(path)`           | 扫描指定路径                   |
| `scanVolume(volumeId)` | 扫描指定存储卷                 |
| `cancelScan(volumeId)` | 取消指定卷的扫描               |
| `cancelAllScans()`     | 取消所有进行中的扫描           |
| `getScanState()`       | 获取当前扫描服务状态           |
| `getActiveTasks()`     | 获取活跃任务列表               |
| `registerCallback()`   | 注册扫描进度回调               |
| `unregisterCallback()` | 取消注册扫描进度回调           |

#### F08: 多卷并行扫描

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 多个 USB 设备同时插入时，各自创建独立的扫描任务，并行执行扫描 |
| **并发策略** | 最多 2 个并发 Worker 线程，防止 USB IO 竞争导致系统卡顿<br>超出部分排队等待 |
| **隔离性** | 每卷使用独立的 `volume_id`，数据完全隔离，一个卷的失败不影响其他卷 |

#### F09: 指定路径扫描

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 客户端应用（如文件管理器）可通过 AIDL `scan(path)` 方法，触发对某个特定目录的扫描 |
| **场景**  | 用户打开文件管理器某个文件夹 → 应用请求扫描该文件夹 → 快速返回该路径下的媒体文件 |
| **约束**  | 扫描结果仍按 `volume_id` 归属，与全量扫描数据合并去重         |

#### F11: 定时增量刷新

| 项        | 说明                                                         |
| --------- | ------------------------------------------------------------ |
| **描述**  | 服务端后台定期对所有 READY 状态的存储卷执行增量扫描，确保缓存数据与磁盘一致 |
| **周期**  | 默认 30 分钟，可通过配置文件调整                              |
| **策略**  | 使用 WorkManager `PeriodicWorkRequest` 实现，系统自动省电调度 |
| **约束**  | 1. 仅对 READY 状态的卷执行<br>2. 如该卷正在扫描中则跳过本次<br>3. 仅在设备充电或电量 > 50% 时执行（车载场景: 发动机运行中） |

## 6. 范围边界

### 6.1 本期范围 (In-Scope)

- ✅ 内部存储 + USB + SD 卡多卷管理
- ✅ 音频 / 视频 / 图片三类型全量 + 增量扫描
- ✅ USB/SD 热插拔自动响应
- ✅ ContentProvider + AIDL 双通道 API（兼容 MediaStore 字段）
- ✅ 开机自启动与扫描状态恢复
- ✅ 系统 priv-app 签名部署
- ✅ 定时后台增量刷新
- ✅ 扫描进度回调通知
- ✅ 按类型/按卷的查询过滤

### 6.2 本期不做 (Out-of-Scope)

- ❌ 修改 Android Framework 层
- ❌ 网络流媒体 / DLNA / SMB 网络存储索引
- ❌ DRM 保护内容的元数据解析
- ❌ 云端同步或远程数据库
- ❌ 用户自定义封面 / 标签编辑（只读）
- ❌ 播放能力（本项目仅提供索引数据，不播放）
- ❌ 文件转码 / 格式转换
- ❌ 蓝牙设备文件浏览
- ❌ iPod / iPhone USB 连接协议解析

## 8. 风险分析

| 编号 | 风险项                     | 影响等级 | 可能性 | 缓解措施                                                     |
| ---- | -------------------------- | -------- | ------ | ------------------------------------------------------------ |
| R01  | USB IO 竞争导致系统卡顿    | 高       | 中     | Worker 数量限制 ≤ 2，大容量设备分批间隔扫描                   |
| R02  | 数据库损坏导致全部数据丢失 | 高       | 低     | `fallbackToDestructiveMigration()` 自动重建 + 自动重新扫描    |
| R03  | 不同供应商裁剪行为不一致   | 中       | 低     | 不依赖 Framework 层 MediaScanner，自建独立数据库              |
| R04  | 海量文件导致 ANR           | 中       | 中     | 所有 IO 异步执行；前台服务保活；大容量设备暂停策略              |
| R05  | ROM 存储空间不足           | 低       | 低     | 预估 50000 文件数据库 < 20 MB；监控剩余空间                   |
| R06  | 系统升级后兼容性问题       | 中       | 低     | 最小 API 28；不依赖隐藏 API；Google Play 审核变化不敏感       |
| R07  | OTA 升级后数据丢失         | 高       | 中     | 开机自动恢复扫描；数据库不存 `/data/data/` 临时路径           |

## 9. 约束条件

| 约束           | 说明                                                         |
| -------------- | ------------------------------------------------------------ |
| **架构约束**   | 不修改 Android Framework 层，以独立 APK 形式运行              |
| **部署约束**   | 部署在 `/system/priv-app/`，需要系统签名                      |
| **兼容约束**   | 数据库 Schema 核心字段对齐 MediaStore，降低生态应用接入成本    |
| **进程约束**   | 独立进程 `android:process=":scanner"`，崩溃不影响其他应用      |
| **权限约束**   | 最小权限原则，仅申请扫描所需权限；API 访问使用 `signature` 保护 |
| **平台约束**   | 最低 API 28 (Android 9.0)，覆盖主流车载系统版本               |

### 附录 B: 与原生方案对比

| 维度               | 原生 MediaScanner            | 本方案                        |
